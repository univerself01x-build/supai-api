# 速派 · 构建方法论

> 从本项目提炼。后续任何产品开卷即用。

---

## 一、交流与决策

### 对话模式
```
你的角色：提出方向、质疑假设、做最终决策
我的角色：提供信息、做判断、执行、坦诚说不

正确："不要。单文件 HTML 就够了。"
错误："好的主人，TypeScript 方案很优秀。"
```

### 决策框架
```
技术选型 = 复杂度匹配
  → 一个页面 = 不需要 React
  → 500行业务 = 不需要 LangChain
  → 不是"这个技术好不好"，是"这个技术配不配"

每次选型问：
  1. 真的需要吗？（90% 的答案：不）
  2. 标准库能搞定吗？（优先）
  3. 加一个依赖解决了什么问题？
```

## 二、构建流程

```
PRD（1页）→ API契约 → 代码 → Harness → Red Flags → 发布

PRD:       做什么、不做什么、给谁用
API契约:   输入/输出/错误格式
代码:      先写后端再写前端。薄壳 + 厚核
Harness:   每个端点 ≥ 3 个测试
Red Flags: 对照清单逐条检查
发布:      一条命令，一个人能跑
```

### 文件命名
```
引擎代码       engine.py (不是 matching_service.py)
产品文档       PRD-WEB.md (不是 product-requirements-document.md)
测试           harness.py (不是 test_suite.py)
规范           PLAYBOOK.md (不是 standards-and-practices.md)
短、能懂、不装。
```

## 三、技术选型法则

### 永远优先
1. 标准库
2. 单文件
3. 零构建

### 永远跳过
1. 框架（除非页面 ≥ 10 个）
2. 构建工具（除非团队 ≥ 3 人）
3. 状态管理库（除非状态 ≥ 20 个）

### 本项目的选择及原因
| 决策 | 选择 | 跳过 | 原因 |
|------|------|------|------|
| 后端 | Python 标准库 | FastAPI/Django | 1个API |
| 前端 | 原生 HTML/JS | React/TypeScript | 1个页面 |
| AI | 直接调 API | LangChain | AI够聪明了 |
| 数据 | JSON 文件 | PostgreSQL | MVP阶段 |
| 通信 | JSON over POST | GraphQL | 1个端点 |

## 四、架构铁律

### 薄壳厚核
```
薄壳 = HTML / API endpoint（参数校验 + 转发）
厚核 = engine.py（全部业务逻辑）
壳可以换，核不动。
```

### 双通道
```
人类 → handle_message()   → {human: 卡片}
Agent → a2a_*()           → {agent: JSON}
同一个引擎，两种格式。
```

### 三层分离
```
入口层（Bot/Web）→ 引擎层（engine.py）→ 输出层（card_*）
单层挂了不影响其他。
```

## 五、品味原则

### 前端
```
暗色主题 / 系统字体 / 无框线 / 流式反馈
空状态有引导 / 错误有解释 / 加载有过程
移动端优先 / 不横向滚动
```

### 后端
```
15 行能搞定的事不写 50 行
函数名 = 做完什么（handle_message, not process_request）
错误信息 = 用户能看懂的中文
```

### 测试
```
harness.py 一键跑
不是"覆盖率 90%"——是"改完代码敢不敢直接上线"
```

## 六、反模式（本项目踩过的坑）

| 坑 | 正确做法 |
|------|------|
| 把逻辑塞进 platform_hints | 逻辑放 Skill，hint 只一句话 |
| 依赖模型做排版 | 引擎渲染格式，模型只传递 |
| 聊天 Bot 做复杂交互 | Bot 只通知，交互走 Web |
| 改配置要重启 | Skill 和 engine 改了秒生效 |
| 滴滴命名照搬 | 有自己的语言（资深/专业/速拍） |
