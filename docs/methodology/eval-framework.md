# 速派 · 评测框架 v1.0

> 2026 共识：80% AI 项目失败是因为没有评测框架。不是模型不行。
> 参考：eBay Mercury eval pipeline、YC Startup School "measure what matters"

## 为什么需要

Harness 24/24 只测代码不炸。不测产品对不对。Eval 体系测后者。

## 三层评测

```
L1 · 单元层（每次改代码）    → Harness 24/24
L2 · 功能层（每次 Phase 完）  → 匹配准确率 + 响应时间
L3 · 产品层（每次客户测试）   → 接受率 + 闭环率 + NPS
```

## L2 功能层指标

| 指标 | 定义 | 当前值 | 目标 |
|------|------|--------|------|
| 匹配准确率 | skill 匹配命中 / 总匹配 | 未知 | >80% |
| 空匹配率 | match 返回 0 结果 / 总匹配 | 未知 | <10% |
| 响应时间 | POST /match → 返回 JSON | <1s (本地) | <2s (生产) |
| Tiebreaker 率 | Top2 分差 <10 / 总匹配 | 未知 | <20% |

## L3 产品层指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 匹配接受率 | 客户确认 / 匹配推送 | >70% |
| 摄影师确认率 | 摄影师确认 / 推送通知 | >50% |
| 完整闭环率 | 双方确认 / 项目创建 | >50% |
| 操作耗时 | 输入需求→摄影师确认（分钟） | <15min |

## 追踪方式

- Concierge Test 阶段：峰哥手动记录到 `docs/ops/concierge-test-log.md`
- Phase 2+：store.json 加 `eval_events` 字段，自动记录每次匹配/确认/拒绝
- 每周复盘：Claude 跑 `scripts/eval-report.py` 生成趋势

## 评测触发

| 时机 | 测什么 |
|------|--------|
| 每次 harness | L1 |
| 每次 Phase 结束 | L2 |
| 每次客户测试 | L3 |
| 每周日 | L3 趋势复盘 |
